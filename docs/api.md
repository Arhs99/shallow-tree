# REST API

HTTP API for retrosynthetic analysis and scoring. Wraps the existing `Expander` class — no changes to core search logic.

## Setup

```bash
# Install API extras (--no-remove keeps existing packages like tensorflow)
poetry lock --no-update
poetry install -E api --no-remove

# If tensorflow was installed via conda and got removed, reinstall it:
conda install -y tensorflow-gpu=2.8.0 -c conda-forge
```

**Note:** TensorFlow is installed via conda (not poetry) for GPU support. Running `poetry install` without `--no-remove` can uninstall it. Always use `--no-remove` when adding extras incrementally.

## Running

```bash
# Basic (local models, 1 worker)
SHALLOWTREE_CONFIG=config.yml searchapi

# Remote models (TF Serving via gRPC)
SHALLOWTREE_CONFIG=config_remote.yml searchapi

# Multi-worker production
SHALLOWTREE_WORKERS=4 SHALLOWTREE_CONFIG=config_remote.yml searchapi
```

### Environment variables

| Variable | Default | Description |
|----------|---------|-------------|
| `SHALLOWTREE_CONFIG` | `config.yml` | Path to config YAML |
| `SHALLOWTREE_HOST` | `0.0.0.0` | Bind address |
| `SHALLOWTREE_PORT` | `8000` | Bind port |
| `SHALLOWTREE_WORKERS` | `1` | Number of uvicorn workers |
| `SHALLOWTREE_API_KEY` | unset | API key for authentication (see below) |

### Worker scaling

**Local models (`config.yml`):** Use **1 worker only**. Each worker loads TF and tries to allocate GPU memory — multiple workers will fail with CUBLAS allocation errors.

**Remote models (`config_remote.yml`):** Workers only do CPU work (DFS + cache). TF Serving handles GPU inference. Scale workers freely.

| Scenario | Workers | Config |
|----------|---------|--------|
| Dev / PoC | 1 | `config.yml` (local models) |
| Production (8 CPU) | 4-8 | `config_remote.yml` + TF Serving |
| HPC (100+ CPU) | 16-64 | `config_remote.yml` + TF Serving + Redis |

## Endpoints

### `GET /health`

Returns 200 if the Expander is loaded and ready.

```bash
curl http://localhost:8000/health
```

```json
{"status": "ok"}
```

### `POST /search`

Run retrosynthetic search on a batch of SMILES.

**Request body:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `smiles` | `list[str]` | required | 1-128 SMILES strings |
| `depth` | `int` | `2` | Max DFS depth (1-10) |
| `scaffold` | `str \| null` | `null` | SMARTS pattern for context search |
| `include_routes` | `bool` | `false` | Include synthetic routes in response |
| `include_building_blocks` | `bool` | `false` | Include building blocks in response |

**Example:**

```bash
curl -X POST http://localhost:8000/search \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $SHALLOWTREE_API_KEY" \
  -d '{"smiles": ["CCO", "c1ccccc1"], "depth": 2, "include_routes": true}'
```

**Response:**

```json
{
  "results": [
    {
      "smiles": "CCO",
      "score": 1.0,
      "route": {"1": [["CCO => CC=O.O", "Reduction"]]},
      "building_blocks": null
    },
    {
      "smiles": "c1ccccc1",
      "score": 1.0,
      "route": {},
      "building_blocks": null
    }
  ],
  "metadata": {
    "elapsed_seconds": 0.523,
    "molecules_processed": 2,
    "molecules_invalid": 0,
    "depth": 2
  }
}
```

### Context search (scaffold-aware)

Pass a `scaffold` SMARTS to use `context_search()` instead of `search_tree()`:

```bash
curl -X POST http://localhost:8000/search \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $SHALLOWTREE_API_KEY" \
  -d '{
    "smiles": ["Clc1ccccc1COC5CC(Nc3n[nH]c4cc(c2ccccc2)ccc34)C5"],
    "scaffold": "[*]c1n[nH]c2cc(-c3ccccc3)ccc12",
    "depth": 2,
    "include_routes": true,
    "include_building_blocks": true
  }'
```

### `GET /docs`

Interactive Swagger UI (auto-generated by FastAPI).

## Architecture

```
                        +--- Worker 1 (Expander) --> gRPC --> TF Serving (GPU:8500)
Client -> FastAPI ------+--- Worker 2 (Expander) --> gRPC --/
         (port 8000)    +--- Worker N (Expander) --> gRPC --/
                        +--- all workers <-> Redis (port 6379)
```

- Local config: models loaded in-process, no external dependencies
- Remote config: models served via TF Serving Docker container (gRPC:8500)
- Redis (optional): cross-worker and cross-request cache sharing

## Authentication

Set `SHALLOWTREE_API_KEY` to require an API key for `/search` requests. If unset, no auth is required (for local use).

```bash
# Generate a key
searchapi-genkey
# e.g. outputs: Abc123...xYz

# Start server with auth enabled
SHALLOWTREE_API_KEY=Abc123...xYz SHALLOWTREE_CONFIG=config.yml searchapi

# Requests must include the key
curl -X POST http://localhost:8000/search \
  -H "X-API-Key: Abc123...xYz" \
  -H "Content-Type: application/json" \
  -d '{"smiles": ["CCO"], "depth": 2}'
```

`/health`, `/docs`, and `/openapi.json` are always accessible without a key.

**For external collaborators:** enable the API key and serve behind HTTPS (e.g. nginx or caddy) so the key isn't sent in plaintext.

## SMILES validation

Invalid SMILES are silently skipped. The `molecules_invalid` count in metadata reports how many were rejected. If ALL SMILES are invalid, the endpoint returns 422.

## Verifying correctness

Scores from the API should match `searchcli` output for the same SMILES:

```bash
# CLI
echo 'CCO' | searchcli --config config.yml --depth 2

# API
curl -s -X POST http://localhost:8000/search \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $SHALLOWTREE_API_KEY" \
  -d '{"smiles": ["CCO"], "depth": 2}'
```
